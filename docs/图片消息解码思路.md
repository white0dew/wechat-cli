# 图片消息解码思路

本文档记录 `wechat-cli` 后续支持微信图片 `.dat` 解码的实现思路，当前仅为设计说明，不代表功能已经完成。

## 当前现状

- 聊天记录中的图片消息在 `history --media` 下，只会尝试定位到微信本地的 `.dat` 文件路径。
- 这一步目前属于启发式匹配，能返回候选文件，但不能保证和当前消息一一精确对应。
- 项目里已经有 `decoded_image_dir` 配置项，但还没有真正使用它来输出解码后的图片文件。
- 收藏中的图片当前仍然只显示为 `[图片收藏]`，没有做本地图片定位或解码。

相关代码位置：

- `wechat_cli/core/messages.py`
- `wechat_cli/core/config.py`
- `wechat_cli/commands/favorites.py`

## 目标

后续希望支持以下能力：

1. 在聊天图片消息上返回解码后的真实图片路径，而不只是 `.dat` 路径。
2. 优先输出到 `decoded_image_dir`，避免污染微信原始数据目录。
3. 解码失败时，回退显示原始 `.dat` 路径，而不是直接丢失信息。
4. 为后续 `favorites` 图片支持预留同样的解码能力。

## 基本方案

### 1. 先复用现有图片定位逻辑

继续使用 `wechat_cli/core/messages.py` 中现有的媒体定位逻辑，先拿到候选 `.dat` 文件：

- 路径形态通常为 `msg/attach/<hash>/YYYY-MM/Img/*.dat`
- 当前已经支持按 `chat_username` 的 MD5 限缩搜索范围

这一步仍然是入口，不需要重写。

### 2. 对 `.dat` 做图片格式识别和解码

微信图片 `.dat` 常见处理方式是单字节 XOR。可以通过已知图片头反推出 key。

优先尝试的图片格式头：

- JPEG: `FF D8 FF`
- PNG: `89 50 4E 47`
- GIF: `47 49 46 38`
- BMP: `42 4D`
- WebP: `52 49 46 46`，并额外检查偏移 8 处是否为 `57 45 42 50`

思路：

1. 读取 `.dat` 文件前几个字节
2. 假设目标格式头已知，计算 `xor_key = encoded_first_byte ^ magic_first_byte`
3. 用同一个 `xor_key` 校验后续几个字节是否也能匹配该格式头
4. 一旦命中，就对整文件逐字节 XOR 解码
5. 按识别出的格式扩展名写入输出目录，如 `.jpg`、`.png`

### 3. 输出到独立目录

建议把解码结果统一写到：

- `cfg["decoded_image_dir"]`

输出文件名建议包含足够稳定的信息，避免覆盖，例如：

- 原始 `.dat` 文件名
- 或 `local_id + create_time`
- 或两者组合

### 4. 命令层展示策略

建议行为：

- `history` 默认仍显示 `[图片]`
- `history --media`：
  - 如果成功解码，显示解码后的真实图片路径
  - 如果只能定位 `.dat`，显示 `.dat` 路径
  - 如果完全找不到，保留当前的 `local_id` 回退

如果后续要增强 JSON 输出，建议额外返回结构化字段，而不是只拼接字符串，例如：

- `media_path`
- `decoded_path`
- `media_status`

## 需要注意的问题

### 1. 不是所有 `.dat` 都一定能直接按单字节 XOR 成功解码

应当允许失败，并保留回退路径。

### 2. 当前图片定位不是精确绑定消息

现在的实现只是按月份目录取候选文件，后续如果要提高准确性，需要进一步研究微信图片消息与落盘文件之间的对应关系。

### 3. 避免重复解码

如果同一个 `.dat` 已经成功解码过，应优先复用已有输出文件，避免每次 `history --media` 都重复写磁盘。

### 4. 注意输出目录清理策略

解码结果属于缓存，后续可考虑：

- 永久缓存
- 基于文件名去重
- 增加清理命令

## 建议实现顺序

1. 增加一个独立的 `.dat -> 图片文件` 解码函数
2. 在 `history --media` 链路里接入该函数
3. 增加最小验证样例和回退逻辑
4. 再考虑是否把同样能力扩展到 `favorites`

## 验证建议

至少覆盖以下情况：

- JPEG 图片能正确解码
- PNG 图片能正确解码
- 无法识别的 `.dat` 会安全回退
- 已存在的解码结果会被复用
- `history --media` 在成功/失败两种情况下输出都合理
